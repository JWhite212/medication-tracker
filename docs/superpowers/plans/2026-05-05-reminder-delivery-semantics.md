# P2 - Reminder Delivery Semantics

> **Status:** in progress
> **Branch:** `feat/reminder-delivery-semantics` (stacked on `feat/email-typed-results`)
> **Owner:** Jamie

**Goal:** Make `reminder_events` a real per-attempt record instead of a dedupe shim. A row that exists no longer means "successfully sent" - it means "an attempt was made". Add per-channel status, retry-after-delay, and an attempt counter so failed sends can recover and observability is real.

**Architecture:** Replace `tryRecordReminderEvent` (insert + bool) with a claim/complete pair. The claim is an `INSERT ... ON CONFLICT (dedupe_key) DO UPDATE SET ... WHERE status = 'failed' AND attempt_count < MAX AND last_attempt_at < NOW() - RETRY_DELAY RETURNING id, attempt_count`. If RETURNING returns a row, the caller owns the send for that attempt. After dispatch, `completeReminder` UPDATEs the row with per-channel statuses and the overall status. `sendPushNotification` is changed to return a typed `PushResult` so the dispatcher can branch the same way it does for email.

**Tech Stack:** Drizzle migration on Neon Postgres, existing email/push modules, vitest mocks.

## Schema additions

```sql
ALTER TABLE reminder_events
  ADD COLUMN status text NOT NULL DEFAULT 'pending',
  ADD COLUMN email_status text NOT NULL DEFAULT 'not_configured',
  ADD COLUMN push_status text NOT NULL DEFAULT 'not_configured',
  ADD COLUMN attempt_count integer NOT NULL DEFAULT 1,
  ADD COLUMN last_error text,
  ADD COLUMN last_attempt_at timestamptz NOT NULL DEFAULT now();

CREATE INDEX reminder_events_retryable_idx
  ON reminder_events (status, last_attempt_at)
  WHERE status = 'failed';
```

Backfill: existing rows are pre-fix successes, so the migration sets them to `status = 'sent'`, `email_status = 'sent'`, `push_status = 'not_configured'` (we cannot reconstruct historical push outcomes).

## Retry policy

- `MAX_ATTEMPTS = 3`
- `RETRY_DELAY_MS = 30 * 60 * 1000` (30 minutes)

Constants live in `dispatch.ts`. Documented but not env-tunable - a follow-up PR can pull them from settings if needed.

## Status semantics

| Field         | Values                                              |
| ------------- | --------------------------------------------------- |
| `status`      | `pending` \| `sent` \| `failed`                     |
| `emailStatus` | `not_configured` \| `pending` \| `sent` \| `failed` |
| `pushStatus`  | `not_configured` \| `pending` \| `sent` \| `failed` |

Transition rules:

- Insert row at `status = 'pending'`, channel statuses set to `pending` for channels that will be attempted, else `not_configured`.
- After dispatch:
  - `status = 'sent'` if any configured channel succeeded.
  - `status = 'failed'` if at least one channel was configured and all configured channels failed.
  - `status = 'sent'` if no channels were configured (best-effort, nothing to retry).

## Risks / notes

- This PR changes the public shape of `sendPushNotification`; callers inside the repo are limited to `reminders.ts`, but a future caller must use the new return type.
- The migration is additive; rolling back drops only the new columns.
- The retry index uses a partial predicate (`WHERE status = 'failed'`) so it stays small even as the table grows.
